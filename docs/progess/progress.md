# Collectly 项目进度报告

> **更新日期**：2026-04-24
> **项目定位**：个人私有化部署的 AI 知识收藏与检索管家
> **核心目标**：跨平台收藏内容的统一解析、结构化存储、智能语义检索与高效学习管理

---

## 一、项目概述

### 1.1 项目背景

Collectly 旨在解决 AI 从业者（聚焦大模型、Agent、RAG、多模态四大领域）在跨平台学习中的核心痛点：信息来源分散（抖音、小红书、公众号、B站、知乎、CSDN 六大平台）、收藏内容过载、检索效率低下、回看困难，导致大量有价值的学习资源"收藏即沉没"。

### 1.2 项目定位

- **个人私有化部署**：每个用户独立部署后端，数据与服务完全隔离
- **单用户系统**：无需用户注册/登录/鉴权，部署即用
- **双部署模式并行**：方案A 本地部署+内网穿透 / 方案B 云服务器公网部署，用户自由二选一
- **全平台前端**：React 响应式 H5，一套代码同时适配 PC 浏览器和手机浏览器/微信 H5

## 二、技术架构

### 2.1 整体架构

```
用户（浏览器/手机 H5）
    ↓ HTTPS
Nginx / 内网穿透
    ↓
前端（React 18 + TypeScript 静态资源）
    ↓ HTTP API
后端（FastAPI + Uvicorn）
    ├── 平台解析服务（TikHub API）
    ├── LLM 服务（DashScope / qwen-plus）
    ├── 搜索引擎（SQLite 关键词 + 向量检索）
    ├── 学习管理器（SQLite）
    └── 内容管理器（SQLite）
```

## 四、已实现功能详细说明

### 4.1 前端功能

#### 页面路由
- `/` - 首页仪表盘：统计卡片、领域分布、最近添加
- `/collect` - 收藏页：链接输入 + 知识卡片网格展示
- `/search` - 检索页：搜索框 + 筛选面板 + 结果列表
- `/learn` - 学习统计：图表展示学习数据
- `/settings` - 设置页：API Key 和模型配置

#### 核心组件
- **Layout**：响应式侧边栏布局，PC 端固定侧边栏，移动端抽屉式导航
- **LinkInput**：链接输入组件，支持批量添加、平台识别、解析状态展示、自动清空
- **Search**：搜索组件，支持语义/关键词模式切换、多维度筛选面板
- **FilterPanel**：筛选面板，支持领域/平台/难度/内容类型/学习状态筛选
- **KnowledgeCard**：知识卡片，展示标题、摘要、标签、平台、状态
- **KnowledgeList**：知识列表，支持网格/列表视图切换、排序
- **DetailModal**：详情弹窗，展示完整摘要、知识点、标签、笔记编辑、状态切换
- **Dashboard**：仪表盘，统计卡片、领域分布、最近添加
- **Stats**：学习统计，饼图（领域分布）、柱状图（学习状态）、折线图（学习趋势）
- **Settings**：设置页，API Key 配置、多模态模型管理

### 4.2 后端功能

#### API 端点
| 端点 | 方法 | 功能 |
|------|------|------|
| `/api/parse-link` | POST | 解析单个平台链接 |
| `/api/parse-links` | POST | 批量解析平台链接 |
| `/api/save-content` | POST | 保存解析后的内容 |
| `/api/search` | POST | 智能检索内容 |
| `/api/update-learning-status` | PUT | 更新学习状态 |
| `/api/update-tags` | PUT | 更新标签 |
| `/api/update-note` | PUT | 更新笔记 |
| `/api/content/{content_id}` | GET | 获取内容详情 |
| `/api/learning-stats` | GET | 获取学习统计数据 |
| `/health` | GET | 健康检查 |

#### 核心服务
- **PlatformParser**：6 大平台链接解析，集成 TikHub API
- **ContentManager**：内容存储、去重（哈希+URL）、CRUD 操作
- **SearchEngine**：关键词搜索 + 多维度筛选 + 相似内容推荐
- **LearningManager**：学习状态管理、标签管理、笔记管理、统计
- **LLMService**：DashScope SDK 封装，支持摘要/标签/知识点/ASR

#### 异常处理
- 自定义异常体系：AppException、PlatformParseError、ContentNotFoundError、DuplicateContentError、InvalidStatusError
- LLM 服务异常：LLMServiceError、LLMServiceTimeout、LLMServiceValidationError
- 全局异常处理器注册
- 日志记录到本地文件（按日轮转）

## 六、当前挑战

### 6.1 技术挑战

1. **语义搜索精度不足**：当前基于 SQLite LIKE 的关键词匹配无法满足语义检索需求，需要集成向量数据库（Chroma/Qdrant）和嵌入模型
2. **前端认证代码残留**：项目定位为单用户无认证系统，但前端仍保留 Supabase 认证组件和 useAuth hook，需要清理
3. **API 地址硬编码**：前端 `api.ts` 中 `API_BASE_URL` 硬编码为 `http://localhost:8000/api`，不支持部署模式切换
4. **视频内容解析不完整**：抖音/B站视频内容依赖 ASR 字幕，但当前 ASR 调用可能失败，降级方案仅使用标题作为内容
5. **小红书 OCR 缺失**：图片笔记中的文字无法提取，导致内容不完整

### 6.2 体验挑战

1. **缺少操作反馈**：无全局 Toast 通知，操作成功/失败反馈不直观
2. **批量操作缺失**：无法批量选择、标记、删除内容
3. **加载状态不完善**：部分页面缺少加载骨架屏
4. **错误边界不完整**：组件崩溃可能导致整个页面白屏
5. **移动端体验待优化**：部分交互在移动端不够友好

### 6.3 部署挑战

1. **无部署脚本**：缺少 Windows 和 Linux 的一键部署脚本
2. **无内网穿透集成**：未集成 Cloudflare Tunnel/ngrok/frp
3. **无进程守护**：未配置开机自启和崩溃重启
4. **环境变量文档缺失**：缺少完整的 `.env` 配置说明

## 九、关键决策记录

### 9.1 已做决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 前端框架 | React 18 + TypeScript | 需求文档要求，生态成熟 |
| 后端框架 | FastAPI | Python 异步支持好，性能优秀 |
| 数据库 | SQLite | 个人项目，无需独立数据库服务 |
| LLM 服务 | DashScope（阿里百炼） | 国内可用，性价比高 |
| 平台解析 | TikHub API | 统一接口，降低开发成本 |
| 链接输入方式 | 手动粘贴 | 避免爬虫风控和法律风险 |
| 认证方案 | 绕过认证（mock 用户） | 个人项目无需复杂认证 |

### 9.2 待决策事项

| 事项 | 选项 | 建议 |
|------|------|------|
| 向量数据库 | Chroma / Qdrant | Chroma（本地轻量级，无需独立服务） |
| 嵌入模型 | DashScope 嵌入 / BGE | DashScope 嵌入（统一 API，减少依赖） |
| OCR 服务 | PaddleOCR / 阿里云 OCR | PaddleOCR（本地运行，免费） |
| 内网穿透 | Cloudflare Tunnel / ngrok / frp | Cloudflare Tunnel（免费，稳定） |
| 进程守护 | Windows 服务 / NSSM | NSSM（将 Python 注册为 Windows 服务） |
